# Frontier 验收测试用例 (Acceptance Testing)

> 版本: v1.0
> 日期: 2026-04-09
> 范围: 基于当前 Docker Compose 标准启动的 Frontier 全栈系统
> 定位: 站在最终用户和业务交付角度，验证系统是否满足核心业务目标
> 与功能测试的区别: 功能测试逐组件验证 UI 行为；验收测试验证端到端的用户目标是否可达成

## 设计原则

1. 每条用例对应一个完整的业务目标，而非单个 UI 交互
2. 步骤描述用户意图，不绑定具体按钮文案或 CSS 选择器
3. 预期结果以业务价值衡量（"用户能完成 X"），而非技术指标
4. 前置条件包含完整的环境与数据状态
5. 用例之间相互独立，可单独执行

## 环境要求

- 启动方式: `docker compose up -d --build`（仓库根目录）
- 验收前提: `http://localhost:8000` 可正常访问，LDAP 登录可用
- 测试账号: 至少一个普通用户 (`testuser/test123`)、一个管理员 (`admin/admin123`)
- 测试数据: 至少一个项目含可用 Agent，该 Agent 可正常响应聊天

## AT-001: 新用户首次登录并完成一次 AI 对话

| 项目 | 内容 |
|---|---|
| **业务目标** | 一个从未使用过系统的用户，能够从零开始登录、发现可用的 AI Agent、发起对话并获得有意义的回复 |
| **优先级** | P0 |
| **前置条件** | 系统已启动；存在一个可用项目，其中至少有一个已配置且可正常响应的 HTTP Agent（如 Demo HTTP Agent）；测试用户账号存在但从未登录过（或清除 localStorage 模拟首次使用） |
| **步骤** | 1. 打开 `http://localhost:8000`，系统应展示登录页面 <br> 2. 使用有效账号登录 <br> 3. 登录后观察首页，应看到 Dashboard 上可用的 Agent 列表 <br> 4. 选择一个 Agent 开始对话 <br> 5. 输入一条消息并发送 <br> 6. 等待 AI 回复完整返回 |
| **验收标准** | ✅ 用户无需任何额外指引即可完成从登录到获得 AI 回复的全过程 <br> ✅ Dashboard 展示了至少一个可点击的 Agent <br> ✅ 聊天消息以流式方式逐步呈现，最终完整显示 <br> ✅ 整个过程中无白屏、无 500 错误、无未捕获异常 |
| **状态** | Draft |

## AT-002: 项目管理员创建项目并配置可用 Agent

| 项目 | 内容 |
|---|---|
| **业务目标** | 拥有 Workbench 权限的用户能够创建一个新项目、在其中配置一个连接到外部服务的 Agent，并使该 Agent 对团队成员可用 |
| **优先级** | P0 |
| **前置条件** | 系统已启动；当前用户有 Workbench 权限；示例 Agent 后端（http-example 容器，端口 8080）运行正常 |
| **步骤** | 1. 从侧边栏进入 Workbench <br> 2. 创建一个新项目（使用合法的小写名称） <br> 3. 进入新建项目的 Agent 管理页面 <br> 4. 添加一个 HTTP 类型的 Agent，端点填写 `http://http-example:8080` <br> 5. 保存 Agent 配置 <br> 6. 回到 Dashboard，检查新 Agent 是否出现 <br> 7. 点击该 Agent 并发送一条测试消息 |
| **验收标准** | ✅ 项目创建成功，出现在 Workbench 项目列表中 <br> ✅ Agent 配置保存成功，出现在项目的 Agent 列表中 <br> ✅ 新建的 Agent 在 Dashboard 可被当前用户发现 <br> ✅ 通过该 Agent 发送消息能收到正常的流式回复 |
| **状态** | Draft |

## AT-003: 用户跨会话保持对话历史连续性

| 项目 | 内容 |
|---|---|
| **业务目标** | 用户的对话历史在关闭浏览器后不会丢失，重新登录后能继续之前的对话 |
| **优先级** | P0 |
| **前置条件** | 系统已启动；用户已登录并与某 Agent 进行过至少一轮对话（包含用户消息和 AI 回复） |
| **步骤** | 1. 在某个 Agent 的对话中发送一条有辨识度的消息（如"请记住数字42"） <br> 2. 确认收到 AI 回复 <br> 3. 关闭浏览器标签页 <br> 4. 重新打开 `http://localhost:8000` 并登录 <br> 5. 从侧边栏历史记录中找到之前的对话 <br> 6. 点击进入该对话 |
| **验收标准** | ✅ 侧边栏历史记录中能找到之前的对话条目 <br> ✅ 点击后完整加载之前的消息历史，包括用户消息和 AI 回复 <br> ✅ 消息内容与发送时一致，无截断或乱码 <br> ✅ 可以在该对话中继续发送新消息 |
| **状态** | Draft |

| **优先级** | P1 |

## AT-005: 项目启用审批流后 Agent 变更需经审批才能生效

| 项目 | 内容 |
|---|---|
| **业务目标** | 当项目启用了审批工作流后，对 Agent 的修改不会立即生效，需要指定的审批者批准后才能应用 |
| **优先级** | P1 |
| **前置条件** | 系统已启动；存在一个项目，当前用户为 owner；项目中已有至少一个可用 Agent；存在至少一个可指定为审批者的用户 |
| **步骤** | 1. 以项目 owner 身份进入项目设置的 Approval 页签 <br> 2. 启用审批工作流，并添加一个审批者 <br> 3. 进入 Agent 管理，修改某个 Agent 的配置（如修改描述或端点） <br> 4. 保存修改 <br> 5. 观察保存后的提示信息 <br> 6. 进入 Change Requests 页面查看待审批请求 |
| **验收标准** | ✅ 启用审批流后保存提示变为"已提交审批"而非"已保存" <br> ✅ Agent 的当前配置保持不变（修改未立即生效） <br> ✅ Change Requests 页面中出现待审批的变更请求 <br> ✅ 变更请求包含准确的修改内容摘要 |
| **状态** | Draft |
